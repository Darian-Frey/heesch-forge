# M2.6-followup-A — Bounded-DLX + SAT partial-state seed

*Cut: 2 May 2026, on commit `4fdbcc0` of `main` (M4.3) plus the
M2.6-followup-A implementation that follows.*

PROPOSAL §5 Layer 2's open follow-up — promised in the M2.7
Phase-2 paper draft as the implementation that would make the
"task-class-based hybrid" architecture actually run fast — has
landed as code. This document records what shipped, what the bench
numbers look like on the M0.4 dataset at n ∈ {7, 8}, and the two
architectural findings that fall out.

## What ships

Five concrete pieces, all on `main`:

1. **DLX node-count budget.** `src/dlx/dlx.{hpp,cpp}` gains
   `set_node_budget(size_t)`, `last_budget_exhausted()`. The
   search-tree recursion checks `last_explored_ > node_budget_`
   on entry and flips `keep_going` to false if exceeded. All
   18 prior DLX tests pass.

2. **Oracle budget plumbing.** `src/corona/oracle.{hpp,cpp}` adds
   `set_node_budget` / `last_budget_exhausted` that delegate to
   DLX. All three solve sites (`find_completion`,
   `count_completions`, `for_each_completion`) read the budget on
   entry and report exhaustion on exit.

3. **`compute_heesch_bounded`.** `src/corona/heesch_solver.{hpp,cpp}`
   adds the budgeted variant. Returns
   `BoundedHeeschResult { complete, result, optional<CoronaState>
   partial_state, ... }`. On `complete = false`, `partial_state`
   carries the deepest hole-free chain DLX built before the budget
   ran out — that's the seed that travels to sat.

4. **Heesch-sat seed support.** `src/sat/src/heesch.h` gains
   `addSeedPlacement(xform_t, level)` and `solveFromSeed(...)`.
   Seed placements are emitted as forced unit clauses by
   `getClauses()`. `src/sat/src/sat.cpp` gains a `-seed <file>`
   CLI flag that loads a `CoronaState` v1 file (omino-only),
   advances `level_` via `increaseLevel()` to the seed depth,
   registers placements, and calls `solveFromSeed` instead of
   `solve`. Build wires `src/corona/libcorona.a` into the sat
   binary so the loader can use `CoronaState::read`.

5. **Orchestrator + harness.**
   `benchmarks/hybrid/bounded_dlx.cpp` runs `compute_heesch_bounded`
   per shape and writes a CSV plus per-shape `.cstate` files for
   the budget-exhausted ones. `benchmarks/hybrid/run_bounded_joint.py`
   reads the CSV, accepts the DLX answer for `complete=1` shapes,
   and invokes `sat -seed` for the rest, recording wall and
   correctness against the M0.4 dataset.

## Bench numbers — n = 7

```text
budget   ok    by_path                          wall(dlx+sat)
1 000    3/3   {dlx: 1, sat-no-seed: 2}         291.1 ms
10 000   3/3   {dlx: 1, sat-no-seed: 2}         592.8 ms
100 000  3/3   {dlx: 1, sat-no-seed: 2}       3 652.1 ms
1 000 000 3/3  {dlx: 3}                       11 195.9 ms
```

(`sat-no-seed` rather than `sat-seed` because at these tight
budgets DLX exhausted before completing even level 1, so the
partial state has depth = 1 = level-0 only and the orchestrator
skips `-seed` to dodge a corner-case bug; see "Known issues"
below. The underlying numbers are unaffected — sat does the same
work either way when the seed has no level ≥ 1 placements.)

## Reading the numbers — two findings

### (1) Tighter budget = both faster AND more correct

The conventional CEM-style intuition is that more budget is more
search and more search is better answers. On the heesch
bounded-joint, **the opposite holds**: the lowest budget (1 000
nodes) produces both the lowest wall and the highest correctness.
At budget = 1 000 000, the bounded-joint loses 3 of the 20 n = 8
answers despite spending 30× the wall.

This is M2.4's known limitation in disguise:
`compute_heesch`'s greedy Hh walkback can converge on a
hole-introducing chain at depth Hh+1 even when a different chain
at the same level would extend further hole-free. The M2.4 paper
draft listed three n = 8 shapes where this happens; the M2.6 PoC
closed all three by sat fallback. With the bounded-joint at high
budget DLX runs to completion on those shapes — **and gets the
wrong answer** because the greedy walkback is the bug, not the
search-tree size.

The architectural conclusion that falls out: *DLX-when-complete is
not the same as DLX-is-correct.* The bounded-joint architecture
needs an extra signal — perhaps a sat-cross-check on an
n%-sample of complete-DLX answers — to be production-correct at
loose budgets. At tight budgets, sat fallback handles the gap.

### (2) Subprocess overhead dominates at small n

SAT-only on n = 7 + n = 8 takes ~0.7 s wall total (per
`/usr/bin/time -v`). Bounded-joint at budget = 1 000 takes ~3.0 s.
The 2.3 s gap is dominated by per-shape subprocess overhead:
~16 `sat -seed` invocations on n = 8 plus ~2 on n = 7 plus the
bounded-DLX subprocess plus Python orchestration overhead. At
larger n where each sat call takes seconds, the relative
overhead shrinks. (Empirically validating that requires
extending the bench to n ∈ {11, 12, 17}; future work, not
M2.6-followup-A.)

A future iteration that fuses bounded-DLX and sat into a single
in-process binary should win on small n too — the spec is on
file, the implementation is straightforward.

## Comparison with M2.6 PoC

M2.6's `run_joint.py` "naive joint" (DLX-on-every-shape +
SAT-on-disagreement) reported **159 s** on n ∈ {7, 8}. The
M2.6-followup-A bounded-joint at budget = 1 000 lands at
**~3 s** — a 53× speedup vs naive joint on the same dataset,
matching the architectural prediction in M2.6's writeup
(estimated 3–5 s at 100 ms / per-shape time budget).

| approach | n = 7+8 wall | correctness |
|---|---:|---|
| SAT-only | 0.7 s | 23/23 |
| bounded-joint (budget=1K) | 3.0 s | 23/23 |
| naive joint (M2.6 PoC) | 159 s | 23/23 |
| DLX-only (M2.4) | 11 s | 20/23 |
| bounded-joint (budget=1M) | 91 s | 20/23 |

(SAT-only and bounded-joint(1K) are both perfect; bounded-joint
is 4× slower in absolute wall but the gap collapses if either
moves in-process or sweeps to larger n.)

## Known issues

**`-seed` with depth-1 CoronaState (level-0-only) reproducibly
fails on certain shapes** with `Tried to look up non-existent cell
<x,y> in clause generation`. The orchestrator works around this
by skipping `-seed` when `partial_depth ≤ 1` (degenerate seed,
adds no information anyway). Bug only manifests when DLX
exhausts before completing level 1, so it's purely a corner-case;
the architecture is unaffected. Investigation deferred — the
likely cause is some interaction in HeeschSolver's level-0 setup
when constructed in a context where a seed file was loaded but
no real seeding happens.

**M2.4 greedy-Hh gaps re-emerge at loose budgets**, as documented
in finding (1). The fix is a separate engineering item (full
DFS over Hh chains in `compute_heesch`, or always-sat-verify
in the orchestrator).

## What this delivers for M2.6-followup-A (sub-A of the original M2.6-followup)

PROPOSAL §5 Layer 2's open follow-up promised the bounded-DLX +
SAT partial-state seed implementation. M2.6-followup-A delivers:

- The mechanical pieces: DLX budget, oracle budget,
  `compute_heesch_bounded`, `sat -seed`, end-to-end orchestrator.
- Bench numbers on n ∈ {7, 8}: 53× speedup vs M2.6's naive joint;
  matches sat-only correctness at budget = 1 000.
- Two architectural findings worth landing in the Phase-2 paper's
  "what we learned next" section: (a) DLX-complete ≠ DLX-correct
  due to M2.4's greedy walkback, (b) subprocess overhead
  dominates at small n.

What's NOT in scope: the encoding refactor that decouples
cell-coverage from tile-placement variables for MaxSAT-derived
dense reward. That's M2.6-followup-B, gated on this milestone but
a multi-session lift in its own right. M4.4 (RL dense reward) is
gated on M2.6-followup-B specifically.

## Reproducing

```bash
( cd src/dlx     && make )
( cd src/corona  && make )
( cd src/sat/src && make sat )            # rebuild against patched heesch.h
( cd benchmarks/hybrid && make )

# Single-budget run
python3 benchmarks/hybrid/run_bounded_joint.py \
  --dataset data/kaplan-2022/omino/07omino_0up.txt \
  --budget 1000 \
  --out benchmarks/hybrid/results/m2.6-followup-A-n7-budget1K.csv

# Budget sweep
for B in 1000 10000 100000 1000000; do
  for N in 07 08; do
    python3 benchmarks/hybrid/run_bounded_joint.py \
      --dataset data/kaplan-2022/omino/${N}omino_0up.txt \
      --budget $B \
      --out benchmarks/hybrid/results/m2.6-followup-A-n${N}-budget${B}.csv
  done
done
```
