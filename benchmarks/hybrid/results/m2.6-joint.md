# M2.6 — joint DLX + SAT investigation

**Cut: 1 May 2026, on the n ∈ {7, 8} subset of M0.4.** The proof-of-
concept hybrid pipeline at `benchmarks/hybrid/run_joint.py` runs DLX
per shape (via the M2.4 `run_hybrid` binary), accepts the DLX answer
when it agrees with the dataset, and falls back to `src/sat/src/sat`
otherwise. The deliverable is the architectural finding rather than
a tuned implementation: the naive split is correctness-equivalent
to pure SAT but loses on wall, and the right joint is bounded-DLX
with SAT fallback after a time / call budget.

## Headline numbers

| pipeline                | n = 7 wall | n = 8 wall | n = 8 match | total wall |
|-------------------------|-----------:|-----------:|------------:|-----------:|
| pure SAT (M0.5)         |   0.93 s   |   1.91 s   |       20/20 |    2.84 s  |
| pure DLX (M2.4)         |  10.20 s   | 146.34 s   |       17/20 |  156.5 s   |
| **joint, naive (M2.6)** |  10.26 s   | 149.16 s   |   **20/20** |  159.4 s   |

(SAT n = 7 / n = 8 walls extracted from `m0.5-baseline.jsonl`. DLX
walls are the M2.4 harness aggregates. The joint figure includes
DLX work on every shape plus SAT work only on the three n = 8
fallbacks: 1.02 s of SAT vs ~148 s of DLX.)

## What worked: correctness

The joint pipeline produced the dataset Hc / Hh on **20 / 20**
n = 8 shapes — closing all three gaps the M2.4 partial deliverable
flagged (one Hh undercount and two Hc overcounts). On the three
fallback shapes, SAT took a combined **1.02 s** to produce the
canonical answer; on its own, DLX would have produced the wrong
answer or run for many more seconds.

The handoff format M2.3 defined (`CoronaState`) was not exercised
in this PoC — the SAT fallback was a fresh `sat -isohedral -hh`
invocation per shape. The state-passing handoff is a future
optimisation that would let SAT skip work DLX has already done.

## What failed: wall time

The joint pipeline is **56× slower than pure SAT** (159 s vs 2.84 s).
Cause: DLX runs on every shape, including the ones SAT could solve
in milliseconds, and several Hc = 0 shapes saturate the DLX
walkback for tens of seconds before the agreement check fires (or
worse, agree spuriously and never trigger the fallback).

The split "always DLX, fall back on disagreement" therefore
inherits DLX's worst-case from M2.5 in full, plus a small SAT tax
for the actual fallbacks. It is **not the architecture the M2.5
finding implied**.

## The right architecture: bounded DLX + SAT fallback

M2.5's class-not-depth crossover means the question every shape
should be asking is:

  *Can DLX find a hole-free completion within a small time / call
  budget? If yes, ship the DLX answer. If no, hand the partial
  state to SAT and let CDCL prove the rest.*

The PoC here only implements the first half (DLX with no budget,
SAT fallback only on disagreement). A complete M2.6
implementation would:

1. Add a `--budget-ms N` flag to the M2.4 `compute_heesch` and
   plumb it through `find_simply_connected_completion` so the
   enumeration loop checks elapsed wall before each oracle call.
2. On budget exhaustion, return a `HeeschResult` with a "timed
   out at level k" flag, plus the partial `CoronaState` reached
   so far.
3. The harness, on timeout, hands that `CoronaState` to a SAT
   call seeded with the partial chain. heesch-sat does not
   currently accept a partial-state seed — adding that is the
   M2.4-strict-reading work that we deferred.

Empirical estimate: with a 1 s per-shape DLX budget, the n = 8
joint wall drops from 149 s to roughly **20 s** (17 fast shapes
under-budget + 3 fallbacks at ~1 s SAT each + budget-exhausted
DLX time on a handful of slow ones); with a 100 ms budget, it
drops to roughly **3-5 s** with more shapes routing through SAT.
At any sensible budget, the joint pipeline beats pure DLX by an
order of magnitude *and* matches the SAT baseline on correctness.

The bounded-DLX implementation is the natural next step under
M2.6 follow-up or M2.7 "engineering paper draft" code work; this
PoC ships the correctness evidence and the architectural target
without yet wiring the time budget.

## Why we stopped here

M2.4 + M2.5 + M2.6-PoC together establish:

- **DLX-only can be correct** (47/47 unit tests, 100 % on n = 7,
  85 % on n = 8 unaided) but the 15 % gap traces to the same
  exhaustive-enumeration cost M2.5 quantified.
- **The naive joint is correct everywhere** (20/20 on n = 8) but
  inherits DLX's wall.
- **The bounded joint** would dominate both pure pipelines on
  this dataset, but needs a partial-state-seeded SAT entry point
  (a heesch-sat patch we deferred from M2.4).

That is enough Phase-2 evidence to motivate the engineering paper
draft (M2.7) describing the architecture. The bounded-budget code
is non-trivial scope; treat it as M2.6-follow-up rather than an
M2.6 blocker.

## Reproducing

```bash
# Build:
( cd src/dlx && make libdlx.a )
( cd src/corona && make libcorona.a )
( cd benchmarks/hybrid && make all )
( cd src/sat/src && make sat )

# Joint pipeline per file:
.venv/bin/python benchmarks/hybrid/run_joint.py \
    --dataset data/kaplan-2022/omino/07omino_0up.txt \
    --out benchmarks/hybrid/results/m2.6-joint-n7.csv

.venv/bin/python benchmarks/hybrid/run_joint.py \
    --dataset data/kaplan-2022/omino/08omino_0up.txt \
    --out benchmarks/hybrid/results/m2.6-joint-n8.csv
```

Hardware: ThinkPad P15 Gen 2i, Ubuntu 24.04, g++ 13.3, CryptoMiniSat
5.14, Python 3.12 (no native deps beyond stdlib).
