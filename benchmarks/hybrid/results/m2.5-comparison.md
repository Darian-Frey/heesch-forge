# M2.5 — DLX-only vs CMSat scaling comparison

**Cut: 1 May 2026, on the 23-shape n ∈ {7, 8} subset of the M0.4
regression.** Joins the M0.5 SAT per-shape timings
(`benchmarks/baseline/results/m0.5-baseline.jsonl`) with the M2.4 DLX
per-shape timings (`benchmarks/hybrid/results/m2.4-hybrid-n*.csv`) by
polyomino coordinates and reports per-shape ratios + aggregate
breakdowns. Reproducible via
`benchmarks/hybrid/compare_dlx_sat.py`.

## Headline numbers

23 shapes total (3 heptominoes + 20 octominoes).

| metric        | CMSat (M0.5) | DLX-only (M2.4) | ratio (DLX / SAT) |
|---------------|-------------:|----------------:|------------------:|
| total wall    |       2.84 s |        772.4 s  |      **272× slower** |
| per-shape p50 |      69.5 ms |        250.1 ms |        5.0× slower |
| per-shape p95 |     317.5 ms |       70.4 s    |    7127× slower    |
| per-shape max |     956.6 ms |      623.1 s    |   10,642× slower    |

DLX-only loses on every percentile. The slowest single shape under DLX
runs **6 500× slower** than the slowest shape under CMSat. This was
foreshadowed in `paper/drafts/phase1-engineering-negatives.md` §4
("CryptoMiniSat is already very strong on this formula class") —
M2.5 quantifies it on the corona-state side specifically.

## The crossover is not a depth, it's a class

The M2.5 milestone wording ("identify crossover depth") presumed a
neat function of corona depth; the data instead shows a much sharper
**problem-class** split.

### By target Hc value

| Hc | shapes | DLX-match | SAT total | DLX total | ratio_p50 |
|---:|-------:|----------:|----------:|----------:|----------:|
|  0 |      7 |       5/7 |   42.5 ms | 131,714 ms |  **80.96×** |
|  1 |     16 |     13/16 |  2,794 ms |  640,710 ms |       1.33× |

For Hc = 0 shapes — the dataset's "no hole-free corona-1 exists"
cases — DLX is **catastrophically slower** (median 81×, with the
worst single shape at 623 s vs CMSat's tens of milliseconds). For
Hc = 1 shapes — where some hole-free corona-1 exists — DLX is
roughly the same order of magnitude as SAT.

The cause: when no hole-free chain exists, the DLX-only walkback
must enumerate every corona completion through the M2.2 oracle and
test each for simple-connectedness, eventually giving up after the
exhaustive search. CMSat proves "no satisfiable hole-free chain"
through CDCL conflict learning, which compresses entire branches of
the search tree into a single learned clause. The cost of "proving
unsat" is exactly where SAT is canonically stronger than DLX.

### By target Hh value

| Hh | shapes | DLX-match | SAT total | DLX total | ratio_p50 |
|---:|-------:|----------:|----------:|----------:|----------:|
|  1 |     22 |     18/22 |  1,880 ms |  772,278 ms |     11.36× |
|  2 |      1 |       0/1 |    957 ms |       146 ms |      0.15× |

The Hh = 2 row's "0.15×" is **not a DLX win** — it's the M2.4 Hh
undercount: DLX terminated greedily at Hh = 1 because its level-1
choice was a dead end for level 2, so it never paid the level-2
cost. CMSat correctly found Hh = 2 (and paid for it). Excluding
that case, DLX is uniformly slower.

## What this means for M2.6

The architectural conclusion is now empirical, not just speculative:

- **DLX's competitive regime is "find first valid completion."**
  When a hole-free chain exists and the oracle finds it quickly, DLX
  is roughly comparable to SAT.
- **DLX's disaster regime is "prove no completion exists."** Hc = 0
  determination requires exhaustive enumeration through the oracle;
  for any non-trivially constrained shape, the runtime is dominated
  by failed alternatives.
- **The right hybrid is therefore not a per-corona-depth split** but
  a per-question split. M2.6's joint formulation should hand
  Hh-style "find any completion" queries to DLX (where the oracle's
  exact-cover encoding is a natural fit) and Hc-style "is there a
  hole-free completion" queries to SAT (where CDCL conflict
  learning prunes the search). M2.3's `CoronaState` is already the
  bridge that makes this split feasible.

## Files

- `m2.5-comparison.csv` — per-shape join (one row per shape):
  coords, n, dataset / SAT / DLX Hc and Hh, both wall-times, ratio,
  oracle / SAT call counts.
- `m2.4-hybrid-n7.csv`, `m2.4-hybrid-n8.csv` — DLX harness output.
- `../../baseline/results/m0.5-baseline.jsonl` — SAT baseline.

## Why we did not run n ≥ 9

The M2.4 write-up estimated multi-hour wall on n = 11 / 12 from the
73 s n = 8 worst case. M2.5 confirms the scaling argument
quantitatively: the Hc = 0 octomino at 623 s would project to
several hours on the n = 9 Hc = 0 octominoes (1 390 shapes) and
correspondingly worse on n ≥ 11. Running them would burn a
multi-day GPU-grade compute budget for a result whose direction
(DLX-only is slower than SAT, scaling makes the gap worse) is
already established at the n ≤ 8 sample.

The right scaling investigation under M2.6 is on the **hybrid**
pipeline once it lands, not the DLX-only one.

## Reproducing

```bash
# Build:
( cd src/dlx && make libdlx.a )
( cd src/corona && make libcorona.a )
( cd benchmarks/hybrid && make all )

# Re-collect DLX timings (writes results/m2.4-hybrid-n*.csv):
( cd benchmarks/hybrid && make test-n7 )
( cd benchmarks/hybrid && time ./run_hybrid \
    ../../data/kaplan-2022/omino/08omino_0up.txt \
    > results/m2.4-hybrid-n8.csv )

# Compare against the M0.5 SAT baseline:
.venv/bin/python benchmarks/hybrid/compare_dlx_sat.py \
    > /dev/null    # CSV goes to results/m2.5-comparison.csv
.venv/bin/python benchmarks/hybrid/compare_dlx_sat.py
    # prints the aggregate summary
```

Hardware: ThinkPad P15 Gen 2i, Ubuntu 24.04, g++ 13.3, CryptoMiniSat
5.14, single thread per binary.
