# M2.6-followup-B session 1 — encoding-refactor attempt (failed)

*Cut: 2 May 2026, on commit `ea7a9f1` of `main` (M2.6-followup-A) +
the failed session-1 attempt that follows.*

This document records a session-1 attempt at the
**M2.6-followup-B** encoding refactor (decouple cell-coverage
from tile-placement so MaxSAT can extract a partial-corona
score), what went wrong, and what session 2 should do
differently. The attempt was reverted from `main`; nothing here
is in the tree.

## What was tried

The plan as proposed before the session: introduce per-cell-per-
level `coverage[c, k]` decision variables; rewrite Blocks B/C/D
of `getClauses` in `src/sat/src/heesch.h` to use `coverage` in
place of the per-cell-only `cell_var[c]`. The plain CMSat path
should remain semantically equivalent to the original; the
session-2 MaxSAT path would then drop Block D and expose
`(coverage[c, k])` weight 1 as soft unit clauses for RC2.

Concrete edits attempted:

1. Added `coverage_vars_: std::vector<std::map<size_t, var_id>>`
   indexed by `cell_index`, plus `getCoverageVariable(p, level)`
   accessor with lazy allocation.
2. Removed `const` from `getClauses` (lazy alloc breaks the
   contract); all call sites are member methods so no external
   API change.
3. Rewrote Block B as `tile[t, k] -> coverage[c, k]` for each
   shape cell c (was `tile[t, k] -> cell_var[c]`).
4. Rewrote Block C as `coverage[c, k] -> ∨_t tile[t, k]` per
   level (was `cell_var[c] -> ∨_t ∨_k tile[t, k]` global).
5. Rewrote Block D as `tile[t, k] -> coverage[c_halo, k+1]` for
   c_halo in halo of t (was `tile[t, k] -> cell_var[c_halo]`).
6. Added a pre-pass at the top of `getClauses` to allocate every
   `coverage[c, k]` Blocks B/D will reference, then call
   `solv.new_vars(delta)` for the additional vars before any
   clauses are emitted.

Encoding "solv.new_vars adds, not sets" was caught and fixed
mid-session.

## What broke

`benchmarks/kaplan/run_regression.py --sizes 7,8,11,12` against
the refactored binary:

```text
overall: match=22 mismatch=152 inconclusive=0 isohedral=0
```

vs. the original encoding's `match=174 mismatch=0`. n = 7 (3
shapes) all matched. n = 8 had 1 mismatch (Hh = 2 in dataset,
sat returned (1, 1)). n = 11 + 12 contributed the rest of the
152 — the same Hh = 2 -> sat = (1, 1) pattern dominates, with a
few `Hc = 2 -> sat = (1, 1)` cases scattered in.

The shape `1 0 2 0 1 1 0 2 1 2 1 3 2 3 3 3` (n = 8, dataset
Hc = 1 Hh = 2) is the smallest reproducer — original gives
`~ 1 2 0`, refactor gives `~ 1 1 0`.

## Why the refactor is wrong

The semantic difference between the original Block C and the
refactored Block C':

- **Original**: `cell_var[c] -> ∨_t ∨_{any level} tile[t, level]`.
  A cell, once forced true, can be satisfied by *any* tile at
  *any* level that covers c.
- **Refactored**: `coverage[c, k] -> ∨_t tile[t, k]` per level.
  A cell labelled at level k must be covered by a *level-k* tile.

In principle these are equivalent: with Block E (no overlap),
only one tile at one level can actually cover a cell, and Block
F (adjacency) constrains which level that tile can be at. The
per-level form is "tighter" but should give the same SAT/UNSAT
result on the same inputs.

In practice, Hh detection's culled-adjacency path
(`extendLevelWithTransforms(level_-1, cloud_.adjacent_culled_)`
followed by a fresh solve with `allow_holes=true`) breaks under
the refactor. The 152 mismatches all show Hh under-reported,
suggesting the refactored encoding rejects a SAT solution that
the original accepts when culled adjacencies are in play.

The most likely culprit (untested, leaving for session 2):
**Block D's per-level `tile[t, k] -> coverage[c_halo, k+1]`
forces a particular level-(k+1) tile to cover c_halo, but
culled-adjacency level-(k+1) tiles may not be the same set as
the regular adjacency level-(k+1) tiles, and the refactored
Block C' may rule out solutions that the original's per-cell
disjunction permitted via cross-level assignments**. The
original encoding's looseness — `cell_var[c]` shared globally,
disjunction over all levels in Block C — quietly absorbed a
case my refactor doesn't.

## What session 2 should do differently

Three options, listed by increasing surgical cost:

1. **Keep the per-cell `cell_var[c]` exactly as-is, ADD
   `coverage[c, k]` as a parallel set of variables, and make
   the WCNF backend emit `(coverage[c, k])` softs that are
   linked via `coverage[c, k] -> cell_var[c]` (one direction,
   hard).** This gives MaxSAT a relaxation point without
   touching any of the existing logic. The cost: doubles the
   number of cell-related variables, and the soft-encoding
   semantics need careful thought (covering a cell at any level
   would satisfy the soft, not just at level_).

2. **Keep Block C global as in the original, but rewrite Block
   D using per-level coverage.** This combines the loose
   Block C (any-level disjunction) with per-level halo coverage,
   which preserves the original's solution space while still
   exposing per-level coverage variables that MaxSAT can target
   as softs.

3. **Get to the bottom of why Block C' is too restrictive.**
   The 152-mismatch failure says "the refactor rejects a SAT
   solution the original accepts." Building two solvers in
   parallel (one per encoding) and dumping their CNFs side-by-
   side on the n = 8 reproducer would identify the missing
   clause / extra constraint in one pass. Estimated half a day.

I'd recommend (3) before committing to (1) or (2): without
understanding the failure, either alternative might bake the
same bug into MaxSAT under a different name. The reproducer is
small (n = 8, single shape) and the comparison is mechanical.

## What's recoverable from session 1

- The plan structure (variables, blocks, pre-pass, scope split)
  is sound at the high level; the specific Block C rewrite is
  the failure point.
- The build scaffolding (sat.cpp's link against libcorona is
  unrelated to this encoding work and is M2.6-followup-A's
  legacy, fine).

This document is committed for the historical record so a future
session can refer to "we tried per-level Block C and Hh broke;
investigate before retrying."
