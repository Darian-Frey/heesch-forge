# M4.2 — RL pilot scaffold + random-policy baseline

*Cut: 2 May 2026, on commit `470e252` of `main` (M4.1) plus the
M4.2 implementation that follows.*

PROPOSAL §5 Layer 3's M4.2 deliverable is "GNN policy network
(PyTorch Geometric); baseline random-policy reward distribution."
This document records what landed and what the random-policy
distribution looks like.

(including 3 sat-binary round-trip checks).

## GNN forward-pass smoke test

A canonical P-pentomino observation (5 cells, 10 directed edges,
9 boundary positions) drives the GNN to produce 9 add-logits and
1 stop-logit:

```text
cells=[(0, 0), (0, 1), (0, 2), (1, 0), (1, 1)]
node_features.shape=(5, 6)
edge_index.shape=(2, 10)
boundaries.size=9
add_logits.shape=(9,)
stop_logit=0.0353
```

This is the M4.2 contract: an untrained `HeeschPolicy` instance
forward-passes correctly on any canonical observation. M4.3 will
add the cross-entropy training loop on top of this scaffold.

## Random-policy reward distribution

Two seeds of the random baseline:

### `n_max = 8`, 1 000 trajectories, seed 0

```json
{
  "n_max": 8,
  "n_trajectories": 1000,
  "elapsed_s": 5.883,
  "trajectories_per_s": 169.97,
  "oracle_cache_size": 297,
  "reward_distribution": {
    "-0.5": 3,
    "0.0": 979,
    "1.0": 18
  },
  "length_mean": 7.06
}
```

- 0.3 % hole-introducing terminations.
- 97.9 % evaluate to H_c = 0 (tiler / inconclusive at maxlevel 7).
- 1.8 % evaluate to H_c = 1.
- **No H_c ≥ 2** — Kaplan's catalogue says first H_c = 2 polyomino
  is at n = 9, so n_max = 8 hitting zero matches expectation
  exactly.

### `n_max = 12`, 5 000 trajectories, seed 0

```json
{
  "n_max": 12,
  "n_trajectories": 5000,
  "elapsed_s": 616.392,
  "trajectories_per_s": 8.11,
  "oracle_cache_size": 2867,
  "reward_distribution": {
    "-0.5": 147,
    "0.0": 4611,
    "1.0": 241,
    "2.0": 1
  },
  "length_mean": 9.4144
}
```

- 2.94 % hole-introducing terminations (rises with n_max — more
  cells, more hole-closing opportunities).
- 92.22 % H_c = 0.
- 4.82 % H_c = 1.
- **0.02 % H_c = 2 — one shape in 5 000 trajectories.**

This is the **floor** the trained policy must beat: the M4.5 exit
criterion (recall ≥ 1.0 of Kaplan's H ≥ 2 set at n ≤ 12 within
B = 10⁵ trajectories) requires the policy to find every Kaplan
H ≥ 2 polyomino at n ≤ 12, not just one. Kaplan reports 2
polyominoes at n = 9 with H_c = 2, 1 at n = 10, plus 39+10+67 at
n = 11..13 (catalogue at `data/kaplan-2022/`). At the random
floor of 1 H_c = 2 per 5 000 trajectories, hitting all of those
within B = 10⁵ would be coincidence. A trained policy needs to
beat random by at least an order of magnitude.

## Throughput notes

`n_max = 8` ran at 170 traj/s; `n_max = 12` ran at 8.1 traj/s.
The slowdown is dominated by the sat solver: shapes at n = 12
take longer to classify (more cells means more SAT clauses, more
corona-completion variables). Memoisation on canonical cells caps
total sat invocations at ~300 (n = 8) and ~2 900 (n = 12). The
oracle's per-call timeout is 30 s; M4.2 saw zero timeouts in
either run.

For M4.3's training loop (cross-entropy on top-K trajectories),
the trajectory throughput at n = 12 will dominate wall time:
~10⁵ trajectories at 8 traj/s ≈ 3.5 hours per evaluation pass.
This argues for either (a) a trajectory-pool architecture that
batches sat calls and caches more aggressively, or (b) the
in-process pybind11 binding flagged in M4.1 §9. Both are M4.3 /
M4.4 work, not M4.2.

## What's not done in M4.2

- **No training.** `HeeschPolicy` is initialised with random
  weights; it forward-passes but does not learn. M4.3 adds the
  cross-entropy loop.
- **No batched evaluation.** The baseline runs one trajectory at
  a time. Batched env / batched GNN inference / batched sat is
  M4.3+ work.
- **No replay buffer.** Trajectories are not stored. M4.3 wires
  this in.
- **No D₄ replay augmentation.** Storage / sampler infrastructure
  doesn't exist yet; the augmentation is an M4.3 add.

These are scoped out of M4.2 deliberately: the milestone is
"baseline reward distribution," not "trained policy."

## Reproducing

```bash
cd src/rl
uv sync --dev                                      # one-time, ~2 GB
uv run pytest -q                                   # 17 tests, ~7 s
uv run python -m heesch_rl.baseline --n-max 8  --trajectories 1000   # ~6 s
uv run python -m heesch_rl.baseline --n-max 12 --trajectories 5000   # ~10 min
```

The `--n-max 8` run is fast enough for development feedback. The
`--n-max 12` run is the baseline reported above; it requires the
patched sat binary (`bitgrid<256>` from M3.2-followup) at
`src/sat/src/sat`.
