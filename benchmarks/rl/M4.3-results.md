# M4.3 — Deep cross-entropy training loop

*Cut: 2 May 2026, on commit `b5228cb` of `main` (M4.2) plus the
M4.3 implementation that follows.*

PROPOSAL §5 Layer 3's M4.3 deliverable is "Implement deep
cross-entropy training loop (Wagner 2021)." This document records
both the implementation and what the training behaviour looks like
on a realistic pilot configuration. **The loop works mechanically
but does not lift the policy past the random-discovery baseline at
our scale; this is M4.1 §7's "reward-signal failure" mode and it
points clearly to M4.4 as the load-bearing next milestone.**

## Implementation

Three new modules in `src/rl/heesch_rl/`:

| file | role |
|------|------|
| `trajectory.py` | `Step` / `Trajectory` dataclasses, D₄-augmented replay sampling, top-K elite selection. |
| `trainer.py` | Wagner-style CEM loop: sample N trajectories, score by terminal reward, top-K elites, train via supervised imitation, repeat. |
| `training_runner.py` | CLI entry point (`python -m heesch_rl.training_runner`). |

The trainer follows Wagner 2021 §3 directly: each generation is a
self-contained supervised-learning phase with no PPO / actor-critic /
value-learning machinery. The loss is the negative log-likelihood
of elite (state, action) pairs under the *current* policy. D₄
augmentation per M4.1 §6 is applied at imitation-loss time: each
elite step is rotated by a random D₄ element before the GNN
re-encodes the resulting state.

The action sampling at trajectory-generation time uses
`torch.multinomial` over the softmax of `[add_logits, stop_logit]`,
with the Stop logit `-inf`-masked when below `n_min`.

Three pytest tests exercise the loop:
`test_sample_trajectory_runs`, `test_select_elites_orders_by_reward`,
`test_train_smoke` (a 2-generation × 6-trajectory run with a stub
oracle, verifying the loop completes and returns the expected
stats shape). The full pytest suite is now 20 tests passing in
`~2 s` (sat round-trip tests excluded if binary not built).

## Reading the numbers

- **`rmax = 1.0` every generation** — H_c = 1 is the search ceiling
  the policy actually reaches; H_c = 2 is never found. Not even
  once across 3 000 trajectories at n_max = 10, where the random
  baseline at n_max = 12 found 1 in 5 000.
- **`rmean ≈ 0.03 ± 0.02`** — the H_c = 1 hit rate fluctuates
  around the same band as the random baseline at n_max = 8
  (1.8 %). No clear upward trend. The CEM update is not pulling
  the policy toward higher-reward regions of shape space.
- **`unique_terminals` ≈ 150 each generation** — the policy is
  not collapsing to a single output. Diversity is preserved.
- **Loss decreases slowly** (`2.2 → 2.0`) — gradients flow, the
  imitation objective is being optimised, but not in a direction
  that translates to better search.

This is M4.1 §7's **"reward-signal failure"** mode, distinguished
from "mode-collapse" (which would show falling unique-terminals)
and "search-coverage failure" (which would show high precision but
low recall — both impossible to measure here since H_c = 2 is never
seen). The policy is mechanically updating but the reward signal
is too sparse for CEM to extract a discovery gradient.

## Why this was the expected risk

PROPOSAL §86 motivated MaxSAT-derived dense reward (largest-partial-
corona score) precisely because vanilla SAT yields a yes/no signal
per corona depth. Sparse terminal H_c is a yes/no-per-trajectory
signal at our scale, where H_c ≥ 2 occurs in 10⁻⁴ of polyominoes.
The cross-entropy method needs at least a handful of elite
trajectories with non-trivial reward in each generation; at 200
trajectories per generation and 0.07 % H_c = 2 prior, expected
elites with H_c = 2 per generation is ≈ 0.14, i.e. usually zero.

M4.1 §4.2 explicitly noted: "MaxSAT-derived dense reward deferred
until the encoding refactor (M1.5's open finding) lands."
M4.4 ROADMAP entry: "Reward = MaxSAT-derived score from
Layer-1+2 solver."

The M4.3 result is therefore the *expected* outcome of vanilla CEM
on the heesch reward landscape: it works, it does not blow up, but
it does not lift the search past random because the reward is too
sparse. This is a directly publishable Phase-4 negative result
pointing to M4.4 as the load-bearing next milestone, which is in
turn gated on the M2.6-followup encoding refactor.

## What this delivers for M4.3

ROADMAP M4.3 entry: "Implement deep cross-entropy training loop
(Wagner 2021)."

This document records: implementation landed (`trainer.py`,
`trajectory.py`, `training_runner.py` + tests); pilot run on a
realistic (n_max = 10) configuration completed; behaviour
documented; failure mode classified per M4.1 §7. M4.3 is closed
modulo M4.4 (dense reward swap), M4.5 (pilot evaluation), and the
two decision-gate fallbacks (MCTS-with-priors, imitation warm-start
from Kaplan).

## What's not done

- **MaxSAT-derived dense reward (M4.4).** Blocked on M2.6-followup
  (bounded-DLX + SAT partial-state seed) which is the encoding
  refactor M1.5 surfaced.
- **Pilot evaluation against Kaplan (M4.5).** Cannot run
  meaningfully against the M4.5 exit criterion (recall ≥ 1.0 of
  Kaplan H ≥ 2 at n ≤ 12 within B = 10⁵ trajectories) until M4.4
  lands; the random-policy floor already shows the trained policy
  is at ≈ random rate.
- **Decision-gate fallbacks.** MCTS-with-priors and imitation
  warm-start from Kaplan's H ≥ 1 catalogue are options if M4.4
  also fails to lift the search.
- **Trajectory-pool / batched sat.** The per-generation cost of
  ~30 s at batch 200 is OK for M4.3 but limits M4.5's B = 10⁵
  budget to many hours. M4.4 would benefit from a pooled-oracle
  refactor.

## Reproducing

```bash
cd src/rl
uv run pytest -q                                            # 20 tests, ~2 s
# Smoke run (~7 s wall):
uv run python -m heesch_rl.training_runner --n-max 8 --batch-size 100 --generations 5
# Pilot run (~7 min wall):
uv run python -m heesch_rl.training_runner --n-max 10 --batch-size 200 --generations 15
```
