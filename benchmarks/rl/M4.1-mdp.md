# M4.1 — Phase-4 RL pilot: MDP definition

*Cut: 2 May 2026, on commit `aa19ccc` of `main`.*

This document is the design spec for the Phase-4 reinforcement-learning
pilot. PROPOSAL §5 Layer 3's central claim is that polyomino
construction can be cast as a sequential decision problem and that
Wagner-style deep cross-entropy (Wagner 2021, arXiv:2104.14516) can
recover Kaplan's H = 3 shapes at n = 17 with substantially less
compute than exhaustive enumeration. M4.1 locks the four MDP axes
— state, action, reward, episode termination — precisely enough that
M4.2 can start writing the GNN policy without re-litigating the
spec.

The document does **not** specify the GNN architecture (M4.2), the
training loop (M4.3), reward refinement (M4.4), or the evaluation
protocol's full details (M4.5+). What it does specify is the
contract those later milestones must respect, and the default values
for every axis where a default is reasonable now.

---

## 1. Framing — what M4.1 locks vs defers

**Locked here.**

- The state representation passed to the policy network: a graph,
  with cells as nodes and 4-cardinal cell-adjacency as edges, plus
  a small set of solver-derived per-cell features.
- The action space: add a single square at one of the 4-cardinal
  boundary positions of the current polyomino, plus a "stop"
  action. Add-only; square removal is a decision-gate fallback
  (not default).
- The reward: sparse, terminal-only, equal to *H_c* as returned by
  the v1 CMSat solver from M0–M2. MaxSAT-derived shaping is
  deferred until the encoding refactor (M1.5's open finding) lands.
- Episode termination: stop-action, or |cells| reaches the curriculum's
  current `n_max`, or the partial polyomino becomes invalid (introduces
  a hole, becomes disconnected — the latter is impossible under the
  add-only action space but flagged for the decision-gate variant).
- Symmetry handling: data augmentation by the eight elements of D₄
  applied at experience-replay time (Wagner's approach). The policy
  is **not** equivariant by construction; the network treats each
  augmented sample as an independent trajectory.

**Deferred.**

- Curriculum schedule across n (M5.1's territory).
- MCTS-with-priors fallback (decision-gate territory: triggers if
  pilot fails to recover H = 3 by M4.6's deadline).
- Imitation-learning warm-start from Kaplan's H = 2 catalogue
  (decision-gate territory).
- MaxSAT-derived dense reward shaping (gated on M1.5 / encoding
  refactor; tracked as M4.4).
- Square-removal action (decision-gate territory; flagged below in
  §3.3).
- GNN architecture, optimizer, batch size, replay-buffer policy
  (M4.2 + M4.3).

## 2. State

### 2.1 Internal state

The MDP's underlying state is the partial polyomino — a finite,
edge-connected set of unit-square cells in `Z²`, plus a small flag
indicating whether the policy has emitted "stop". Concretely:

```python
@dataclass(frozen=True)
class MDPState:
    cells: frozenset[tuple[int, int]]   # canonical-form coordinates
    stopped: bool
```

`cells` is normalised to a canonical form before anything else
touches it: translate so the lexicographically smallest cell is at
(0, 0), then pick the lex-smallest representative across the eight
D₄ orbits. This canonical form is the equivalence class the
deterministic SAT solver also operates on, and using it here means
that two trajectories that arrive at the same shape via different
add-orderings hash identically. (Trajectory replay still augments
by D₄ — see §6 — but the *node identity* in episode storage is the
canonical form.)

The `stopped` flag is set once the policy emits the stop action;
no further cells can be added afterwards. The terminal reward is
computed exactly once, at the moment `stopped` flips True or the
episode hits the `n_max` budget.

### 2.2 Observation passed to the policy

The policy does **not** see the raw `MDPState`. It sees a graph
observation `O(s)`:

- **Nodes.** One node per cell in `state.cells`.
- **Edges.** One undirected edge between each pair of 4-cardinally
  adjacent cells. (No diagonal edges; this matches the cell-adjacency
  graph the deterministic solver uses for hole detection.)
- **Node features.** Per cell, a feature vector `f(c) ∈ R^d` with:
  - `(x, y)` of the cell after canonicalisation (2 dims; raw coords
    after canonicalisation, unnormalised — shapes are small enough
    that normalisation is unnecessary at the n ≤ 17 scale).
  - `is_boundary`: 1 if the cell has a 4-cardinal neighbour outside
    `state.cells`, else 0.
  - `boundary_degree`: number of 4-cardinal neighbours outside
    `state.cells` (∈ {0, 1, 2, 3, 4}).
  - `is_corner`: 1 if the cell has exactly two diagonally-opposite
    occupied neighbours and the rest empty (a heuristic that
    Kaplan's H = 3 shapes feature heavily; cheap to compute).
  - `n_normalised`: `|cells| / n_max` — gives the policy a
    horizon signal.
- **Edge features.** None initially. (M4.2 may add a relative-position
  edge feature if empirical results suggest the policy under-uses
  geometric structure; this is an M4.2 ablation, not an M4.1 commit.)

Total per-cell feature dim `d = 6`. This is intentionally shallow:
the GNN should learn most of the structure from the topology, not
from hand-engineered features.

### 2.3 Action-space view of the state

The policy also needs to know which actions are available. We
include the legal actions in the observation as a separate
attribute rather than as graph nodes:

- **`legal_add_positions`**: a list of `(x, y)` boundary cells —
  the 4-cardinal empty neighbours of any occupied cell. Length is
  variable (the boundary grows roughly linearly with `|cells|`).
- **`stop_legal`**: True if `|cells| ≥ n_min` (a small floor; default
  `n_min = 4`, since H ≥ 1 polyominoes start at n = 7 and a one-cell
  polyomino can't usefully be evaluated).

`legal_add_positions` lengths in n ≤ 17 stay under ~40 in practice
(boundary of compact shapes); for irregular shapes that produces
slightly larger boundaries but still well within a single batch
dimension.

## 3. Action

### 3.1 Action types

Two action types in the default add-only design:

1. **`Add(x, y)`** — extend the polyomino by adding the unit square
   at `(x, y)`, where `(x, y) ∈ legal_add_positions`. The new
   `MDPState` is rebuilt with the augmented `cells`, then re-canonicalised.
2. **`Stop`** — emit terminal action. Sets `stopped = True`.

### 3.2 Action representation in the policy head

The action head outputs:

- A categorical distribution over the `len(legal_add_positions) + 1`
  legal actions (the +1 is `Stop`).
- Realised per-step using a softmax over per-position logits plus
  one stop-logit.

The variable-length action space is the standard pattern for GNN
policies on growing graphs. M4.2 will most likely realise this by
running a small MLP head on each boundary node's hidden vector
(producing a per-position add-logit) and on a graph-pool readout
(producing the stop-logit).

### 3.3 Square removal — flagged, not default

Allowing the policy to remove an existing square (rather than only
adding) is the decision-gate fallback if M4.6 fails to recover
Kaplan's H = 3 shapes. It is not default for two reasons:

- **Trajectory length blows up.** With remove allowed, episode
  length is unbounded a priori; we'd need a step budget rather than
  a cell budget.
- **Reward variance.** The terminal reward is computed only at
  episode end; with remove, the agent can spend many steps in
  H = 0 territory before settling.

If we adopt remove, `Remove(x, y)` is restricted to cells whose
removal keeps the polyomino edge-connected. Implementing the
connectivity check on the boundary cells is straightforward (the
4-cardinal cell-adjacency graph stays connected after removing a
non-articulation vertex; we precompute articulation cells via a
single DFS).

### 3.4 Invalid actions

The action space is defined to be exactly the legal actions; the
policy cannot select an action outside `legal_add_positions ∪ {Stop}`
because the softmax is over precisely that set. There is no need
for an "illegal action" mask or penalty. The "stop must respect
n_min" constraint is enforced by simply omitting the Stop logit
from the softmax if `|cells| < n_min`.

## 4. Reward

### 4.1 Default — sparse terminal *H_c*

The episode reward `R(τ)` is computed once, at episode termination:

```python
def reward(state: MDPState, n_max: int) -> float:
    if not state.stopped and len(state.cells) < n_min:
        return -1.0   # below floor; degenerate

    if has_hole(state.cells):
        return -0.5   # hole-introducing; valid polyomino but not
                      # worth a sat call

    if not is_simply_connected(state.cells):
        return -0.5   # disconnected; should be unreachable under
                      # add-only but the check is cheap

    hc = compute_hc_via_sat(state.cells)   # v1 CMSat solver
    return float(hc)
```

The reward shape is therefore:

- `hc = 0`: reward = 0
- `hc = 1`: reward = 1
- `hc = 2`: reward = 2
- `hc = 3`: reward = 3
- (target) `hc ≥ 4`: reward = `hc`

…with `-0.5` for hole-introducing or invalid-shape terminations
and `-1.0` for under-floor terminations.

The decision-gate-eligible alternative — *bound by maxlevel L*
returns `min(hc, L)` rather than `hc` itself — is M4.4's territory.
For the pilot (M4.5), `L = 7` (the default sat maxlevel) is fine;
shapes that would need `L > 7` are statistical curiosities at the
n ≤ 17 scale.

### 4.2 Why not MaxSAT-derived dense reward

PROPOSAL §5 paragraph 86 expressly motivated MaxSAT-derived dense
reward (largest-partial-corona score) as the right RL signal. M1.5
investigated this and found it requires an encoding refactor that
overlaps with M2's planned DLX/SAT handoff (full finding at
`benchmarks/maxsat/README.md`). The bounded-DLX + SAT-seed pattern
from M2.6's deferred follow-up is a natural place for the dense
reward to come from once it's implemented.

For M4.x, sparse terminal *H_c* is the contract. M4.4 swaps it for
the MaxSAT-derived score as soon as the encoding refactor is
available; this swap should be a single function in the reward
module, not a re-design of the MDP.

### 4.3 Reward magnitude / clipping

Sparse rewards in `[−1, ~3]` (Phase-4 pilot range, n ≤ 17, where
H = 3 is the known max) need no clipping. If Phase-5 surfaces an
H ≥ 4 polyomino the reward magnitude grows but the GNN policy's
value head will adapt without architectural change.

The reward is not normalised. Cross-entropy methods (Wagner's
recipe; M4.3's territory) work at the trajectory level — the
top-K trajectories are selected by *raw return*, so the absolute
scale matters less than the relative ordering. We will revisit
normalisation only if M4.5 reports policy-collapse symptoms
(all-zero rewards drowning the cross-entropy signal).

## 5. Episode termination

An episode terminates on the first of:

1. **Stop action.** Policy emits `Stop`; reward computed.
2. **Cell budget reached.** `|cells| == n_max` after the last `Add`.
   `Stop` is then implicit; reward computed.
3. **Invalid state.** `Add` produces a polyomino with a hole or a
   disconnected `cells` set. (Disconnected is impossible under
   add-only; hole-introducing is possible if the new cell closes
   off an empty interior region.) The episode terminates
   immediately with the `-0.5` invalid-shape reward.

The `n_max` budget is curriculum-controlled (M5.1's territory) but
the default for the M4.5 pilot is `n_max = 12` — the smallest n at
which Kaplan's catalogue shows H = 2 polyominoes plus a buffer.
The full pilot escalates to `n_max = 17` per M4.6.

There is no per-step time limit. The `compute_hc_via_sat` call at
terminal is the bottleneck; we cap it via the existing sat
`-maxlevel 7` flag and let the SAT solver itself decide when to
give up.

## 6. Symmetry handling — D₄ data augmentation

Polyominoes are equivalence classes under the dihedral group D₄
acting on `Z²` (rotations by 90°/180°/270° plus four reflections).
Three reasonable handling choices were considered:

- **(a) Canonicalise observations + actions.** Force the policy to
  see only canonical-form states; every action's effect is
  re-canonicalised before the next step. *Clean but constrains the
  policy class — the policy must learn a function on canonical
  forms, which has weird boundary behaviour (small perturbations
  of a near-canonical state can flip its canonical representative).
  Wagner explicitly avoids this.*
- **(b) Replay augmentation by D₄.** Store trajectories in canonical
  form; at training time, sample with one of the eight D₄ elements
  applied. *Wagner's approach. Each canonical trajectory contributes
  up to 8 effective training samples.*
- **(c) Ignore.** Let the policy learn equivariance from data.
  *Wasteful at our scale — we're search-bounded, not data-bounded.*

**Default: (b).** Trajectories are canonical-form for storage; on
each minibatch sample, we draw a random D₄ element `g` and apply
it to (state, action, reward). This is a standard pattern in
RL-for-combinatorics and matches Wagner 2021 §3 directly. The
policy network is **not** equivariant by construction; we do not
attempt to use a D₄-equivariant GNN architecture. This is a
deliberate simplification — equivariant networks are an open
research area and out of scope for the M4.x pilot.

The reward `R(τ)` is invariant under D₄ (the SAT solver's *H_c*
output is too), so reward augmentation is a no-op — we only augment
the (observation, action) pair.

## 7. Validation harness for the M4.5 pilot

The validation criterion is operational: the trained policy must
**rediscover all known H ≥ 2 polyominoes at n ≤ 12** within a
fixed sample budget B (default `B = 10⁵` trajectories), starting
from a random-policy initialisation.

Concretely, after training:

1. Sample `B` trajectories under the current policy (greedy at
   evaluation; stochastic at training).
2. Collect the set `S = {canonical-form of trajectory's terminal
   shape}` for trajectories with terminal reward ≥ 2.
3. Compare `S` against Kaplan's H ≥ 2 catalogue at each `n ∈
   {7, ..., 12}`.
4. Report `precision` (`|S ∩ Kaplan| / |S|`), `recall` (`|S ∩
   Kaplan| / |Kaplan ≥ 2 at n ≤ 12|`), and the trajectory budget
   used to first see each Kaplan shape.

The exit criterion for M4.5 is `recall ≥ 1.0` (every Kaplan H ≥ 2
shape is rediscovered) at `B ≤ 10⁵`. Failure modes M4.5 should
report distinctly:

- **Mode-collapse.** Policy converges on a small set of low-H
  shapes; recall plateaus at < 1.
- **Search-coverage failure.** Policy explores broadly but never
  hits the Kaplan shapes; recall plateaus at < 1, but precision is
  also low.
- **Reward-signal failure.** Most trajectories return −0.5 (hole)
  or `−1` (degenerate); the cross-entropy update has no usable
  signal.

The first two are architecture / hyperparameter problems for M4.2
and M4.3. The third is a problem with this MDP design and would
trigger an M4.1 revision.

## 9. Engineering hooks

The MDP-as-spec needs three concrete bridges to existing project
artefacts:

- **Reward oracle.** `compute_hc_via_sat(cells)` is implemented by
  calling `src/sat/src/sat` as a subprocess (binary already builds,
  patched to `bitgrid<256>` since M3.2-followup). Input: a single
  shape line in heesch-sat's `O? <coords>` format. Output: parse
  the `~ Hc Hh` reply line. Per-call latency at n ≤ 17 is ~ms to
  low seconds; M4.5 will batch where the GNN throughput needs it.
  An in-process C++/pybind11 binding is M4.4 territory if subprocess
  cost dominates.
- **Validation set.** `data/kaplan-2022/` already holds the
  polyomino catalogue parsed by M0.4. The pilot evaluation script
  reads it directly; no new data acquisition needed.
- **Training infrastructure.** Local CPU is fine for M4.5 (n ≤ 12,
  policy network in the few-MB range). Cloud GPU enters at M5.2,
  not M4.x.
