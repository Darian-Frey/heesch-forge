# M1.2a — Kissat backend (single-solver baseline)

Same dataset and harness as M0.5 / M1.1, restricted to sizes 7, 8, 12 (64
shapes) because Kissat's non-incremental design makes the n = 11 batch
take 30+ minutes alone (see "Why n = 11 was skipped" below). Identical
Hc/Hh values on every shape across all three backends — regression
returns 64/64 match in each run; only wall-time differs.

## Three-way headline

| metric                       | CMSat (M0.5) | CaDiCaL (M1.1) | Kissat (M1.2a) |
|------------------------------|-------------:|---------------:|---------------:|
| wall total (n ∈ {7, 8, 12})  |       61.4 s |        119.6 s |        450.3 s |
| ratio vs CMSat               |          1×  |    1.95× slower |    **7.33× slower** |
| per-shape wall, p50          |        0.33 s |          0.44 s |          1.33 s |
| per-shape wall, p95          |        3.58 s |          7.31 s |         32.14 s |
| per-shape wall, max          |        4.53 s |         11.79 s |         33.73 s |
| SAT solve-calls total        |          337 |            336 |            336 |
| SAT conflicts total          |      105,017 |              0 |              0 |
| SAT decisions total          |  (see m0.5)  |              0 |              0 |

`sat_calls` matches to within one across all three backends, confirming
the heesch-sat algorithm itself is unchanged — the only thing that
varies between rows is solver behaviour. Conflict / decision /
propagation counters are zero for CaDiCaL and Kissat because neither
1.7 nor 4.x exposes them programmatically; recorded in
`../README.md` and `src/sat/UPSTREAM.md`.

## Why n = 11 was skipped

Kissat is **non-incremental**: its public C API has no way to add
clauses after `kissat_solve()` has been called, and re-solving the same
instance is undefined. To make Kissat work without rewriting heesch.h,
`kissat_backend.h` buffers every clause in a local vector and rebuilds
a fresh `kissat*` instance — re-feeding **all** prior clauses — on
every `solve()` invocation.

For most heesch SAT calls (`cur_solver`, `iso_solver`, the two
`final_solver`s) this is a no-op: each instance is built once and
solved once, so there is exactly one rebuild and one solve. The
adapter's overhead is zero on these.

`HeeschSolver::iterateUntilSimplyConnected` (heesch.h ~line 875)
breaks that contract by design: it sits in a
`while (solver.solve() == l_True)` loop and adds hole-banning clauses
after every iteration. Each loop iteration triggers a full rebuild.
For n = 11 polyominoes the walkback hits this loop with thousands of
clauses already in the formula and forbids dozens of hole patterns,
giving an O(N²·K) cost that wedged a single live trial at >14 minutes
for one shape before being killed.

The smaller files (n = 7, 8, 12) include shapes that exercise the
walkback loop too — that is where the 33-second tail at p95 / max in
the table above comes from — but the per-shape ceiling is bounded
enough that the 41-shape n = 12 batch finishes in ~7 minutes total
rather than tens of minutes per shape.

## Phase-1 implications

**None of the three modern CDCL competitors beats CryptoMiniSat on
this workload.** The PROPOSAL §5 wording ("replace single-solver SAT
call with a portfolio") needs to be read as *augment*, not *replace*.
A first-to-finish portfolio of CaDiCaL + Kissat + Glucose (with CMSat
excluded) would be strictly slower than CMSat solo at every percentile
in this dataset. Concretely:

- **The portfolio must include CryptoMiniSat.** First-to-finish across
  CMSat + CaDiCaL would, in expectation, just be CMSat with extra
  process-startup overhead, since CMSat wins every percentile here.
  The *only* way a portfolio gains over CMSat-solo is if (a) some
  shape class exists where Kissat or CaDiCaL is faster (we should
  scan for that in M1.2b before committing to architecture), or
  (b) clause sharing between CMSat and the others actually helps.
- **Kissat is a poor fit for heesch-sat as currently structured.**
  Its non-incremental design forces an O(rebuild) per solve in the
  walkback loop. Two paths forward if we want Kissat in the
  portfolio: refactor `iterateUntilSimplyConnected` to use
  one-shot solver instances (closer to Kissat's grain), or use Kissat
  *only* for the non-incremental call sites and route the walkback
  loop through CMSat / CaDiCaL.
- **Phase-1 ≥5× speedup target is not reachable by solver swap
  alone.** This is consistent with M1.1's CaDiCaL finding. The lift
  has to come from M1.3 (BreakID symmetry breaking — orthogonal to
  solver choice) and M1.4 (encoding sweep) rather than from picking
  a different SAT engine.

## Reproduce

```bash
( cd src/sat/src && make clean && make -j$(nproc) )
# Produces sat (CMSat), sat-cadical (CaDiCaL), sat-kissat (Kissat).

python benchmarks/baseline/run_baseline.py --sizes 7,8,12 \
  --sat src/sat/src/sat-kissat \
  --out benchmarks/baseline/results/m1.2a-kissat-baseline.jsonl \
  | tee benchmarks/baseline/results/m1.2a-kissat-baseline-7-8-12.log
```

Hardware: ThinkPad P15 Gen 2i, Ubuntu 24.04, g++ 13.3,
CryptoMiniSat 5.14, CaDiCaL 1.7.4, Kissat (master, source-built and
installed to /usr/local/lib).
