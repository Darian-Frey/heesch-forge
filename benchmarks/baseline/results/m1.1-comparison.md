# M1.1 — CMSat vs CaDiCaL single-solver baseline

Same 174-shape Kaplan-2022 polyomino set (sizes 7, 8, 11, 12), same
`heesch-sat` build, only the SAT backend differs. Identical Hc/Hh values
on every shape (regression: 174/174 match in both runs); the only
question this row answers is *how fast*.

| metric | CMSat (M0.5) | CaDiCaL (M1.1) | ratio (cadical / cms) |
|--------|-------------:|---------------:|----------------------:|
| wall total                  | 256.2 s | 569.2 s | 2.22× slower |
| per-shape wall, p50         |   1.42 s |   2.59 s | 1.82× slower |
| per-shape wall, p95         |   3.53 s |  10.59 s | 3.00× slower |
| per-shape wall, max         |   4.53 s |  15.05 s | 3.32× slower |
| SAT solve-calls total       | 1,047 | 1,046 | ≈ |
| SAT conflicts total         | 612,136 | (n/a) | — |
| SAT decisions total         | 4,010,000 | (n/a) | — |
| SAT propagations total      | 174,222,093 | (n/a) | — |

`sat_calls` matches across backends to within one — same algorithm,
same number of SAT invocations per shape. Conflict / decision /
propagation counters are zero for the CaDiCaL run because CaDiCaL
1.7's public C++ API does not expose them; documented in
`../README.md`.

## Headline

CaDiCaL is *slower* than CryptoMiniSat on this workload, with the gap
widening at the tail (3× at p95, 3.3× at max). At the median the
penalty is only 1.8×. The sat-call count being preserved confirms the
heesch-sat algorithm is unchanged; the gap is purely backend behaviour.

This is **not the Phase-1 result we want**, but it is informative:

- The Phase-1 exit criterion (≥5× speedup, with a <2× decision-gate
  fallback at M1.7) cannot be reached by single-solver swap alone.
  M1.2 (portfolio first-to-finish across CaDiCaL + Kissat + Glucose
  with clause sharing) and M1.3 (BreakID symmetry breaking) need to
  contribute most of the gain.
- Why CMSat looks strong: it is a hybrid CDCL + Gauss-Jordan + SLS
  solver with aggressive inprocessing, generally well-tuned for the
  small-but-numerous SAT calls heesch-sat issues per shape. CaDiCaL's
  strength is on harder, longer-running monolithic queries.
- The portfolio result (M1.2) should at minimum recover the CMSat
  baseline (first-to-finish trivially returns the fastest of the
  three), and ideally beat it via clause sharing on the harder n=11
  / n=12 shapes that drove the p95 tail.

## Files

- `m1.1-cadical-baseline.jsonl` — per-shape JSONL, CaDiCaL backend.
- `m0.5-baseline-7-8-11-12.log` — CMSat console summary.

## Reproduce

```bash
# Build (run once after pulling):
( cd src/sat/src && make clean && make -j$(nproc) )    # produces sat AND sat-cadical

# Re-run baselines:
python benchmarks/baseline/run_baseline.py \
  --out benchmarks/baseline/results/m0.5-baseline.jsonl \
  | tee benchmarks/baseline/results/m0.5-baseline-7-8-11-12.log

python benchmarks/baseline/run_baseline.py --sat src/sat/src/sat-cadical \
  --out benchmarks/baseline/results/m1.1-cadical-baseline.jsonl \
  | tee benchmarks/baseline/results/m1.1-cadical-baseline-7-8-11-12.log
```

Hardware: ThinkPad P15 Gen 2i, Ubuntu 24.04, g++ 13.3, CryptoMiniSat 5.14
(built from source), CaDiCaL 1.7.4 (`libcadical-dev` from `noble/universe`).
